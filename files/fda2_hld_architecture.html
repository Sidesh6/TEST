Custom SWIFT Parser Design — Built In-House
A two-parser library: MT Parser (block-based text) and MX Parser (ISO 20022 XML). Both produce a unified ParsedMessage object handed to the Canonical Model builder. Versioned, pluggable, and testable independently.
MT Parser — SWIFT Block Structure
1.Raw message ingested as byte stream
Encoding detection (UTF-8 / EBCDIC). Strip BOM. Normalise line endings.
↓
2.Block Tokenizer
Split on {1:} {2:} {3:} {4:} {5:} delimiters. Validate block presence / order per message type. Handle {2:I} (input) vs {2:O} (output).
↓
3.Header Parser (Block 1 & 2)
Extract: LT address · session · seq · message type · priority · direction. MsgType registry lookup → loads field definition for that MT.
↓
4.User Header Parser (Block 3)
Parse tag 103 (service id) · 113 (banking priority) · 108 (MUR) · 119 (validation flag: STP/REMIT/COV). Key–value map.
↓
5.Body Field Tag Parser (Block 4)
Line-by-line scan. Tag pattern: :<2-3 char tag>[letter]:<value>. Sequence detection (A/B/C/D). Repeated tag handling (e.g. :57A: multiple). SubField split per tag definition.
↓
6.SubField & Format Validator
Per-tag: length · format codes (n/a/x/c/d/e) · SWIFT character set · conditional presence rules. Errors collected (don't fail-fast).
↓
7.Trailer Parser (Block 5)
MAC · CHK · PDE · DLM · MRF tags. Checksum verification if present.
↓
8.ParsedMTMessage → CanonicalBuilder
Typed model: blocks + field map + seq map + error list. Handed to Canonical Model builder.
MX Parser — ISO 20022 XML
1.Raw XML ingested
Streaming SAX or StAX reader for large messages. Validate well-formedness. Extract AppHdr (head.001) if present.
↓
2.Namespace & Message Type Detection
Extract namespace URI → map to message identifier (e.g. pacs.008.001.10). Version extraction. Route to correct XSD.
↓
3.XSD Structural Validation
Validate against versioned XSD from Schema Registry. Collect all violations (not fail-fast). Schema registry supports hot-add of new versions.
↓
4.JAXB / Custom Unmarshalling
Unmarshal to versioned POJO tree. Code-generated from XSD via xjc OR custom reflection-based for flexibility. Supports pacs · camt · pain · acmt · colr.
↓
5.Business Element Extraction
Navigate DOM for key fields: Cdtr · Dbtr · IntrBkSttlmAmt · EndToEndId · TxId · SttlmDt. Sub-order field extraction (SvcLvl · CtgyPurp · Purp).
↓
6.Code Set Validation
ISO currency codes · country codes · BIC format · IBAN check digit · purpose codes · charge bearer codes from embedded code registries.
↓
7.Repeating Group Handler
CreditTransferTransactionInformation[n] · RemittanceInformation. Cardinality check. Index each transaction for per-txn processing.
↓
8.ParsedMXMessage → CanonicalBuilder
Typed model: AppHdr + Document + txList + error list. Handed to Canonical Model builder.

Error Handling & Resilience Strategy
Every stage produces a sealed result. Failures never silently drop — always captured with full context.
Processing Result Sealed Type
sealed interface ProcessingResult<T> {
  record Success<T>(T payload, MessageContext ctx) implements ProcessingResult<T> {}
  record ValidationError(String errorCode, String field, String reason,
                        RawMessage original, MessageContext ctx) implements ProcessingResult<?> {}
  record SystemError(Throwable cause, String stage,
                    RawMessage original, MessageContext ctx) implements ProcessingResult<?> {}
}

Retry Strategy
• Max 3 retries with exponential backoff
• Idempotency key prevents double-processing
• Per-step timeout defined in pipeline config
• Saga compensating actions for partial failures
• Circuit breaker on downstream endpoints
Observability
• OpenTelemetry trace spans per pipeline step
• correlationId propagated end-to-end
• Structured logs: JSON with traceId + clientId
• Metrics: throughput · latency · error rate per client
• Full audit trail in journal table (immutable)
